# 决策记录 · DECISIONS

每条:背景 / 选项 / 选择 / 代价。只记非显然的判断,供日后回看。

## 1. 子时跨午夜的「诗日」归属

- **背景**:子时是 23:00–00:59,横跨自然日边界。同一夜的 23:30 与次日 00:30 应算「同一个子时之夜」,须显示同一句;而「同日同辰恒显同一句、次日轮换」又要求以日期为轮换键。
- **选项**:(a) 按自然日历日取键——则 23:30 与 00:30 落在不同日、显示不同句,割裂了同一夜;(b) 把 23:00–23:59 归入「次日的子时」,与 00:xx 共用同一诗日键。
- **选择**:(b)。`selectPoem` 中当 `hour === 23` 时把日期 +1 天再折算日序,使一夜的子时(23→次日 0:59)命中同一句,并在每晚 23:00 轮换到新子时诗。
- **代价**:子时的「轮换时刻」是 23:00 而非 00:00,与其余时辰(整点起即当日)略不一致;但换来了「一整夜同一句」的恒常感,值得。已写测试锁定此行为。

## 2. 选诗索引公式

- **背景**:需要一个纯函数、确定性的映射:`(日期, 时辰) → 某一句`。
- **选项**:随机/哈希诗句文本;或 `日序 % n`;或 `(日序 + 时辰序) % n`。
- **选择**:`poemIndex = (dayOrdinal + shichenIndex) % n`。日序用 `Date.UTC(y,m,d)/86400000` 求整数——只为得到随日期 +1 递增、且不受本地时区/夏令时抖动的稳定序号。叠加 `shichenIndex` 让各时辰不在同一天整齐翻页,轮换更耐看,但对「同日同辰恒显同一句」无影响。
- **代价**:当每辰只有 2 句时,轮换是简单的双日交替(可预期而非惊喜)——但这正是产品要的「恒常优先于随机」。

## 3. 预览其他时辰的取值

- **背景**:← / → 或点击左右区可预览别的时辰;预览应展示「今天」那个时辰会显示的句子。
- **选项**:为每个时辰构造一个代表时刻调用 `selectPoem`;或直接复用当前日序另算索引。
- **选择**:用「代表小时表」`[0,1,3,5,…,21]` 构造 Date 后调用同一个 `selectPoem`,复用全部边界逻辑。子时的代表小时特意取 **0 时**(而非 23 时)以避免跨日回滚,保证与其余时辰同属当日。**当前时辰永远取实时 `selectPoem(now)`**,预览只用于非当前时辰,故不会与子时的 23:00 回滚产生可见矛盾。
- **代价**:预览用的是「当前日历日」的轮换,深夜站在子时里跳走再跳回,极端边角可能与实时子时差一句——但预览是短暂态(10s 归位),可接受。

## 4. 测试放在哪

- **背景**:验收要求「子时跨午夜边界正确(附测试)」,而文件结构清单里没有测试文件位置。
- **选项**:新增 `scripts/*.test.js`(超出清单);或把测试内嵌进 `main.js`。
- **选择**:内嵌。`runSelfTests()` 与纯逻辑同处 `main.js`;浏览器加 `?selftest` 运行并浮层显示,Node 亦可 `require('./main.js').runSelfTests(data)`。既满足「附测试」,又不突破「文件结构不超过」清单。
- **代价**:`main.js` 略长;但纯逻辑段本就要随移植带走,测试贴身反而方便。已用 Node 与浏览器双跑,9/9 通过。

## 5. 数据加载方式与 file:// 限制

- **背景**:硬约束「所有诗词数据只存在于 `data/poems.json`,渲染层不得内嵌任何诗句文本」。
- **选项**:(a) `fetch('data/poems.json')`——但浏览器出于安全会拦截 `file://` 的 fetch;(b) 把 JSON 做成 `poems.js` 挂全局(可 `file://` 直开)——但那样文件就不叫 `poems.json` 了,且近似内嵌;(c) 用 `<script type="application/json">` 内联——直接违反「不得内嵌」。
- **选择**:(a) `fetch`。预览需经本地静态服务器(或 GitHub Pages),这正是标准做法。`file://` 直接双击打开会走兜底提示(指向 README),而非白屏。
- **代价**:双击 `index.html` 无法直接渲染,必须起一个静态服务器。换来的是干净的「单一事实来源」。**「断网仍完整渲染」的本意——零外部依赖、字体自托管——已满足**:页面全部请求皆本地,断网不影响。

## 6. 竖排标点与字向

- **背景**:诗句竖排(vertical-rl)。半角逗号 `,` 在竖排中位置尴尬。
- **选择**:诗句一律用**全角逗号 `,`**;`.poem` 用 `text-orientation: mixed`(默认),让汉字直立、全角标点走竖排形(逗号落在字格右上)。落款 meta 用 `〔朝代〕作者　《出处》`,全角/汉字构成,**刻意避开半角间隔号**(半角 `·` 在 mixed 竖排里可能被旋转)。角落时辰名是横排元素,仍可安全用 `·`。
- **代价**:改动了种子数据里的逗号字形(半角→全角),更合中文竖排规范。

## 7. 香篆进度弧

- **背景**:加分项——一条极细弧线示当前时辰内进度;但硬约束禁止「仪表盘感」。
- **选择**:做成一条**浅浅的开口弧**(非闭合圆环),极细(0.8)、低透明,底色一条更淡的轨,已燃部分用 `stroke-dashoffset` 揭示(`pathLength=100`)。开口弧避免了环形量表的仪表感。颜色用淡墨,**绝不用印章红**(红只留给印章)。
- **代价**:进度不是一眼精确可读的刻度——但这本就是意象而非仪表。

## 8. 字体子集与构建机无 Python

- **背景**:要求自托管霞鹜文楷,并用 `fonttools` 写 `scripts/subset_font.py` 子集化到 < 300KB。构建机(Windows)只有商店占位版 python,无 pip/fonttools。
- **选项**:(a) 改用 Node 版子集器现造 woff2——但交付脚本仍须是 Python,那样脚本没被真正跑通验证;(b) 临时引入官方**便携版 Python**(embeddable zip)装 fonttools+brotli,真跑 `subset_font.py`。
- **选择**:(b)。便携 Python 只用于本机生成/验证,不入库;`subset_font.py` 因此是**真跑通并产出过 woff2** 的脚本,而非纸面代码。脚本还加了 `--download` 可自动拉源字体、并强制 UTF-8 输出以适配 Windows 控制台。
- **代价**:源 TTF(约 25MB)不入库,重跑子集需先备好源字体(或加 `--download` 现取)。产物 `fonts/LXGWWenKai-subset.woff2` 为 117.5KB。

## 9. 夜读模式区间与实现

- **背景**:要求 23:00–05:00 自动切夜读(深底暖字,版式不变)。
- **选择**:`isNightHour(h) = h>=23 || h<5`(即 23、0–4 时,覆盖子/丑/寅并及卯前),对应 `body.night`。深浅两套值全走 CSS 变量,**版式完全不变**,仅色彩以 900ms 过渡柔和切换。
- **代价**:5:00 整切回日间可能与真实天光略有出入(各地日出不同)——但固定区间简单、可移植,墨水屏同样适用。

## 10. 时辰印章

- **背景**:全页唯一强调色(印章红)只允许出现在一处小元素。
- **选择**:印章取时辰名首字(子丑寅卯…),**阳文**(纸色字压在朱红底上),小方形置于落款处。经检查全页仅此一处用到印章红。
- **代价**:印章是单字而非完整时辰名(完整名放在角落),但更像一枚真正的闲章。

## 11. 种子诗红线

- **背景**:严禁杜撰/拼接/改写;出处拿不准宁可不用。
- **选择**:24 句全部取广为人知、出处作者齐全的名句(直书时辰意象者优先,如子·「夜半钟声到客船」、亥·「寂寂人定初」)。`poems.json` 顶部 `meta.notice` 明示其为占位,灵魂待主人逐句替换。
- **代价**:意象贴合度高的名句有限,个别时辰在「广为人知」与「意象精确」间取平衡(已在每句 `why` 里说明理由)。

## 12. 交互克制

- **背景**:要求克制——无设置面板、无菜单。
- **选择**:仅 ← / → 键与屏幕左/右点击区预览,中区点击回到当前;预览带「预览」角标,10 秒无操作自动归位。首屏一条极淡提示,数秒后自动隐去、首次交互即消失。
- **代价**:发现性略低(没有显式按钮)——但符合「诗是主角」的克制基调,且有一句提示兜底。

# v1.1 变更记录(2026-07-06)

## 13. 轮换键:按年内第几天 → 按星期几(周循环)

- **背景**:产品化调整。希望「同一星期几的同一时辰恒显同一句」,一周为一个可感知的循环节律(而非按日无限推进)。**本条取代 #1、#2 中关于「日序轮换」的描述。**
- **选项**:(a) 保留日序(dayOrdinal)按日推进;(b) 改用 `Date.getDay()`(0=周日)做 `index = getDay() % 该时辰句数`。
- **选择**:(b)。移除 `dayOrdinal` / `poemIndexFor`,新增 `poemIndexForWeekday(weekday, count) = weekday % count`。**子时跨午夜的滚日逻辑保留**:`hour===23` 时把日期滚到次日再取 `getDay`,使一夜的子时(23:00→次日 00:59)落在同一星期几、同一句——即该子时归入它所引向的清晨那天的星期。
- **代价**:句数不足 7 的时辰在一周内会重复(如 2 句时按 0,1,0,1,0,1,0),这是预期行为、不报错;句数满 7 时每星期几各不相同。测试已从「次日轮换」改为「周循环确定性 + 一周覆盖 + 十二时辰×七星期几均有效」,`?selftest` 与 Node 双跑 **10/10**。

## 14. 曲库改为「候选—审核」制,不直接写入 poems.json

- **背景**:定位为公共产品,曲库需可信、可追溯。要扩到每辰 7 句,但正式库 `poems.json` 不能混入未经人工核对的句子。
- **选择**:新建 `data/candidates.json`(结构镜像 poems.json,每条另加 `confidence` 与 `source_note`)。本轮**只**产出候选,**不改动 `poems.json` 的任何诗句**;晋升由主编 King 逐条人工核对后手动搬入。难凑的时辰(丑·鸡鸣、未·日昳)宁缺毋滥。
- **代价**:曲库暂不满 7 句/辰,需人工介入才能扩容——但换来可信度与可追溯性,符合公共产品定位。

## 15. 推送被本机 git 钩子拦截(流程说明,非代码决定)

- **背景**:`~/.claude/hooks/block-dangerous-git.js` 拦截一切 `git push`。
- **选择**:遵守钩子,不绕过、不保存任何凭证。所有提交在本地完成,推送留给主人在自己的终端执行(见收尾说明)。
- **代价**:远程需主人手动推一次;换来凭证与历史安全。
